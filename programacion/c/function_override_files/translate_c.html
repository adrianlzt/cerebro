Biblioteca defecto primordial a través de LD_LIBRARY_PATH Overriding Default Library via LD_LIBRARY_PATH

Autor: Siro Mugabi Author: Siro Mugabi

Categoría: Ejercicios ELF Category: ELF Exercises

Resumen: Summary:

Un ejercicio en el uso de LD_LIBRARY_PATH para anular la carga de una biblioteca por defecto con una versión privada en tiempo de ejecución. An exercise on using LD_LIBRARY_PATH to override the loading of a default library with a private version at execution time. Este enfoque implica "no selectivo" función de biblioteca interposición. This approach implies "non-selective" library function interposition. Este material es una extensión de la construcción y uso de ELF bibliotecas compartidas tutorial. This material is an extension of the Building and Using ELF Shared Libraries tutorial.

Ejemplo Example

Las instrucciones contenidas en este documento pueden ser realizadas desde cualquier directorio de trabajo conveniente. The instructions contained herein may be performed from any convinient working directory. Para los propósitos de esta guía: For purposes of this guide: 

 $ cd /tmp $ WORKDIR=$PWD

Cree la siguiente biblioteca hola mundo. Create the following hello world library. Para simplificar la ilustración, el nombre de la biblioteca y soname se establecen en el nombre enlazador: To simplify illustration, the library name and soname are set to the linker name: 

 $ cat libfoo.c #include <stdio.h> void lfunc(void) { puts("libfoo"); } $ gcc -Wall -O2 -fPIC -Wl,-soname=libfoo.so -shared libfoo.c -o libfoo.so

Construir el siguiente ejecutable hola mundo para su uso con la biblioteca compartida: Build the following hello world executable for use with the shared library: 

 $ cat prog.c extern void lfunc(void); int main(void) { lfunc(); return 0; }
  • NOTA: Mientras LD_LIBRARY_PATH se puede utilizar durante el lanzamiento del programa para que el enlazador dinámico localizar ./libfoo.so LD_LIBRARY_PATH se utiliza aquí en el contexto de ilustrar de búsqueda de biblioteca de anulación para la función de interposición. NOTE: While LD_LIBRARY_PATH can be used during program launch to enable the dynamic linker locate ./libfoo.so , LD_LIBRARY_PATH is used here in the context of illustrating library search override for function interposition. Para ilustración adecuada, la acumulación ejecutable / install aquí es tal que por defecto compartió la carga de la biblioteca que se hace "transperently" en el lanzamiento del programa de manera que LD_LIBRARY_PATH se puede utilizar cuando anulando la biblioteca por defecto con una versión privada. For proper illustration, executable build/install here is such that default shared library load is done "transperently" at program launch so that LD_LIBRARY_PATH can be used when overriding the default library with a private version.

Para permitir que el enlazador dinámico localizar libfoo.so "transperently" en tiempo de carga, uno de los siguientes enfoques pueden ser utilizados: To enable the dynamic linker locate libfoo.so "transperently" at load-time, one of the following approaches may be used:

  • Insertar un DT_RUNPATH entrada en el archivo de objeto ejecutable. Embed a DT_RUNPATH entry in the executable object file. Tenga en cuenta que este camino de ejecución debe apuntar al directorio que contendrá libfoo.so durante la programación de lanzamiento. Note that this run path should point to the directory that will contain libfoo.so during programing launch. Este enfoque no requiere privilegios de root 1: This approach does not require root priviledges 1 : 

     $ gcc -Wall -O2 prog.c -L. -lfoo -Wl,-rpath=$WORKDIR -Wl,--enable-new-dtags -o prog $ readelf -d prog | grep RUNPATH 0x000000000000001d (RUNPATH) Library runpath: [/tmp]

  • Instale libfoo.so en la ruta de búsqueda enlazador dinámico a través de /etc/ld.so.cache actualización. Install libfoo.so in the dynamic linker search path by way of /etc/ld.so.cache update. Este procedimiento requiere privilegios de root - potenciales permanece desastre si se realiza de forma incorrecta: This procedure will require root priviledges - potential disaster abides if performed incorrectly: 

     $ gcc -Wall -O2 prog.c -L. -lfoo -o prog ### install "./libfoo.so" in "/etc/ld.so.cache" $ sudo sh -c "echo $WORKDIR >> /etc/ld.so.conf" $ sudo ldconfig -v | grep libfoo.so libfoo.so -> libfoo.so

    Cuando se hace con el ejercicio en esta página, no se olvide de restaurar la configuración del sistema, es decir, anteriores borrar el $WORKDIR entrada en /etc/ld.so.conf y volver a ejecutar ldconfig(1) para actualizar /etc/ld.so.cache . When done with the exercise in this page, do not forget to restore previous system settings ie delete the $WORKDIR entry in /etc/ld.so.conf and re-run ldconfig(1) to update /etc/ld.so.cache .

Cualquiera que sea el caso, el comando de inicio será: Whichever the case, the launch command will be: 

 $ ./prog libfoo

Ahora, cree un nuevo libfoo.so en un directorio independiente que exporta a different lfunc() definición, por ejemplo: Now, create a new libfoo.so in a separate directory that exports a different lfunc() definition, for example: 

 $ mkdir newdir && cd newdir $ cat libfoo.c #include <stdio.h> void lfunc(void) { puts("libfoo-newdir"); } $ gcc -Wall -O2 -fPIC -Wl,-soname=libfoo.so -shared libfoo.c -o libfoo.so

Realice el lfunc búsqueda interposición símbolo través LD_LIBRARY_PATH Perform the lfunc symbol lookup interposition via LD_LIBRARY_PATH : 

 $ cd $WORKDIR $ LD_LIBRARY_PATH=${PWD}/newdir ./prog libfoo-newdir

es decir, la búsqueda de la biblioteca, ya que necesita el enlazador dinámico ahora comienza desde ./newdir/ ./newdir/libfoo.so se carga en lugar de ./libfoo.so ie since needed library search by the dynamic linker now begins from ./newdir/ , ./newdir/libfoo.so gets loaded instead of ./libfoo.so : 

 $ LD_DEBUG=all LD_LIBRARY_PATH=./newdir ./prog 2>&1 | egrep '(trying.*foo|symbol.*lfunc|binding.*lfunc)' 26747: trying file=./newdir/tls/x86_64/libfoo.so 26747: trying file=./newdir/tls/libfoo.so 26747: trying file=./newdir/x86_64/libfoo.so 26747: trying file=./newdir/libfoo.so 26747: symbol=lfunc; lookup in file=./prog [0] 26747: symbol=lfunc; lookup in file=./newdir/libfoo.so [0] 26747: binding file ./prog [0] to ./newdir/libfoo.so [0]: normal symbol `lfunc'

y así lfunc símbolo de búsqueda y la unión se satisfizo de ./newdir/libfoo.so lugar de ./libfoo.so and so lfunc symbol lookup and binding gets satisfied from ./newdir/libfoo.so rather than ./libfoo.so .

Notas Notes

Este mecanismo tiene cierta semblence a la de LD_PRELOAD This mechanism bears some semblence to that of LD_PRELOAD . Sin embargo, usar LD_LIBRARY_PATH interponer funciones de biblioteca tiene ciertas restricciones / limitaciones en comparación con el uso de LD_PRELOAD However, using LD_LIBRARY_PATH to interpose library functions has certain restrictions/limitations as compared to using LD_PRELOAD :

  • Soname del biblioteca privada debe mucho una entrada en los DT_NEEDED etiquetas del ejecutable. The private library's soname must much an entry in the DT_NEEDED tags of the executable. Esto significa que la carga de tiempo de ejecución de la biblioteca privada reemplaza el uso de la biblioteca por defecto - y por lo tanto la biblioteca debe soportar por lo menos todas las versiones de () funciones referenciados por la aplicación es decir, la función no selectivo interposición. This means that the execution time load of the private library replaces the use of the default library - and therefore the library must at least support all (versions of) functions referenced by the application ie non-selective function interposition. También significa que el nombre de la librería privada también debe residir en un directorio aparte del nombre de la librería que se overriden. It also means that the soname of the private library must also reside in a directory separate from the soname of the library to be overriden. En este caso: In this case: 

     $ cd $WORKDIR $ tree . ├── libfoo.so # default library ├── newdir │  └── libfoo.so # private library ├── prog # executable

    y: and: 

     $ readelf -d prog | grep NEEDED 0x0000000000000001 (NEEDED) Shared library: [libfoo.so] 0x0000000000000001 (NEEDED) Shared library: [libc.so.6]

    Esta restricción no se aplica con LD_PRELOAD This restriction does not apply with LD_PRELOAD . Su uso en lugar de LD_LIBRARY_PATH permitirá interposición selectiva de símbolos en la biblioteca por defecto, $WORKDIR/libfoo.so por alguna otra biblioteca cuya soname no coincide con ninguno de prog 's DT_NEEDED entradas. Using it instead of LD_LIBRARY_PATH will allow for selective interposition of symbols in the default library, $WORKDIR/libfoo.so , by some other library whose soname does not match any of prog 's DT_NEEDED entries. Esto implica que la biblioteca privada sólo es necesaria en tiempo de ejecución, y que la biblioteca privada ahora puede residir en el mismo directorio que la biblioteca por defecto. This implies that the private library is only required at execution time, and that the private library can now reside in the same directory as the default library.

  • DT_RUNPATH (en lugar de DT_RPATH debe ser utilizado con el fin de búsqueda de la biblioteca con LD_LIBRARY_PATH a prevalecer. DT_RUNPATH (instead of DT_RPATH ) must be used in order for library search with LD_LIBRARY_PATH to take precedence. Esta limitación no se aplica con LD_PRELOAD This limitation does not apply with LD_PRELOAD ; sus valores siempre se buscan primero. its values are always searched first. Ver Usando -rpath un ejemplo para una discusión sobre DT_RPATH frente DT_RUNPATH See Using -rpath , An Example for a discussion on DT_RPATH versus DT_RUNPATH .
